# Self-Coherence: Self-update checksum verification (#161)

## Alpha (type-level clarity)

The new functions have clear signatures and responsibilities:
- `read_file_contents : string -> string` — reads entire file as binary string.
- `Cn_sha256.lookup_checksum : body:string -> binary:string -> string option` — pure parsing of checksums.txt format.
- `Cn_sha256.verify_file_checksum : checksums_body:string -> binary:string -> file_contents:string -> [`Valid | `Mismatch | `No_checksum]` — pure verification.
- `verify_checksum : tag:string -> binary:string -> path:string -> [`Valid | `No_checksums | `Mismatch]` — I/O wrapper.

Named parameters (~tag, ~binary, ~path, ~body, ~checksums_body, ~file_contents) prevent argument transposition.

## Beta (authority surface agreement)

The checksums.txt format is the same format generated by `release.yml` (`sha256sum` output: `<64-hex>  <filename>`). The parsing in `lookup_checksum` handles both single-space and double-space separators via `String.trim` on the remainder after extracting the 64-char hash prefix.

The `do_update` flow now gates on checksum before `--version`, matching the installer's verification order (download → size check → checksum → install). Both surfaces use the same checksums.txt artifact from the release.

## Gamma (cycle economics)

- `cn_sha256.ml`: +24 lines (2 new public functions).
- `cn_agent.ml`: +22 lines net (helper functions + verification gate in do_update, offset by removed inline parsing).
- `cn_sha256_test.ml`: +42 lines (6 new test cases).
- No dead code introduced. Every new line serves an AC.

## Triadic Coherence Check

1. **Do all ACs trace to code?**
   - AC1: `do_update` calls `verify_checksum` before `--version` check. Yes.
   - AC2: `verify_file_checksum` calls `Cn_sha256.hash`. No shell-out. Yes.
   - AC3: `No_checksums` case prints warning, falls through to `--version`. Yes.
   - AC4: `Mismatch` case calls `Sys.remove`, returns `Update_fail`. Yes.
   - AC5: `lookup_checksum` and `verify_file_checksum` in `cn_sha256.ml`, tested in `cn_sha256_test.ml`. Yes.

2. **Does the installer and self-update use the same verification logic?**
   - The installer (shell) reimplements SHA-256 via `sha256sum`/`shasum`. The self-updater uses `Cn_sha256.hash` (pure OCaml). Both parse the same checksums.txt format. The OCaml side is canonical; the shell side is an external tool invocation that produces the same result.

3. **Can a compromised binary still pass?**
   - With checksums: only if the attacker controls both the binary and checksums.txt on the same release. This is a GitHub-hosted supply chain boundary — acceptable.
   - Without checksums (pre-v3.33.0): `--version` check still runs. This is the existing behavior, now explicitly documented as a fallback.

4. **Are there silent failure paths?**
   - `read_file_contents` can raise if file is unreadable. This would propagate as an uncaught exception, causing `Update_fail` via the outer match. The file was just downloaded successfully, so this is unlikely but not silent — it would crash visibly.
   - `curl` failure returns `None`/empty → `No_checksums` → fallback with warning. Not silent.
