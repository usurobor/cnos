**AC1 — dispatching a design-first issue can no longer leave body-hold and label state contradictory.**

- **Fixture reconstructing the exact #614/#633 shape (positive case).** `TestDispatch_ContradictoryShape_BodyCorrectedLabelUntouched` (`src/packages/cnos.issues/commands/issues-dispatch/dispatch_test.go`) builds an issue with `contradictoryBody` (a body ending in `"**Not dispatched** — \`status:ready\`. Design-first; dispatch on explicit operator authorization."`, preceded by a `---` separator — the live #614/#633 shape) plus labels `["dispatch:cell","protocol:cds","status:todo"]` (the label already authorized, body still stale — the exact drifted contradiction, not the "about to dispatch" case). Running `Dispatch(ctx, opts)` with `Apply: true` asserts: `res.BodyChanged == true`, exactly 1 `EditBody` call, the stored body equals `contradictoryBodyWant` (the hold paragraph and its preceding separator stripped), and **zero** label-mutation calls (the label was already correct — a corrected, not a redundant, result). This is an automated unit test with an injectable GitHub-API double (`fakeIssueStore`), not a live-network manual check.
- **Clean issue is a safe no-op (negative case).** `TestDispatch_AlreadyCleanAndTodo_SafeNoOp` builds an issue with no hold phrase and `status:todo` already set. `Dispatch` returns `res.NoOp == true`, and the fixture's mutation counters (`edits`, `removed`, `added`) are all zero — proven, not just claimed.
- **Pure-function coverage of the same two cases**, independent of any GitHub double: `TestReconcileBody_StripsHoldPhrase_BoldWithSeparator` / `_PlainNoSeparator` (positive, two phrasing variants) and `TestReconcileBody_CleanBodyIsSafeNoOp` (negative) in `reconcile_test.go`.
- **Idempotency oracle.** `TestDispatch_Idempotent_SecondRunIsNoOp`: runs `Dispatch` twice against the same stateful fixture (`status:ready` + hold phrase). The first run applies both actions; the second run — observing the post-first-run state — reports `NoOp == true` and issues zero further HTTP-shaped calls. Body, labels, edit count, and label-mutation counts are asserted identical before/after the second run.
- **"One caller-visible operation, no half-applied state" oracle.** `TestDispatch_BodyEditFails_LabelNeverFlipped`: injects an `EditBody` failure via `fakeIssueStore.editErr`. `Dispatch` returns an error whose text contains `"label not touched"`, and the fixture asserts **zero** label-mutation calls occurred and `status:ready` remains present (never flipped to `status:todo`) — i.e. a body-edit failure can never produce the exact contradiction (`status:todo` label + un-cleaned hold phrase) this primitive exists to fix, even though the underlying operation is still two separate HTTP calls (`ghEditIssueBody` then `ghRemoveLabel`+`ghAddLabel`) executed body-first by construction (`dispatch.go`'s `Dispatch`, comment at the body-edit call site).
- **Live-wiring proof (caller-path trace).** `TestLiveDefaults_RouteThroughSharedPrimitives` (`fetch_test.go`) exercises `Dispatch` with a zero-value `Options{}` (no injected fakes) against a real `httptest.Server`, proving the default wiring (`liveGetIssue` → `ghEditIssueBody` → `ghRemoveLabel`/`ghAddLabel`) actually reaches the HTTP layer: exactly 1 GET, 1 PATCH, 1 DELETE, 1 POST captured.
- **Candidate (3), the optional contradiction gate, was NOT built as a separate mechanism.** The detection logic `ReconcileBody` computes internally is the same logic a standalone gate would need; γ's scaffold explicitly allowed "add it if cheap given what you've already built." α judged a separate gate to be net-negative complexity for this cycle: `cn issues dispatch` itself already performs the detection-and-correction on every operator invocation, and a same-cell CI/reconciler gate would need its own selector loop over open issues (a second, wake-adjacent mechanism the scope guardrails' "operator/human-invoked only" framing does not obviously extend to) which was not part of the pinned implementation contract.
