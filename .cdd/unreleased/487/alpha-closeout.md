## §3 Friction notes (Stage 1)

### FN-α-Stage1-1 — Prompt declaration-only banner missed at R0 (the coupling lesson)

γ scaffold §4.3 forbade prompt.md touches (Sub 4 / cnos#483 landed it) and α correctly didn't touch it at R0. But the prompt contained pre-merge `declaration-only` framing that contradicted the post-merge manifest `activation_state: live`. The wake's `prompt:` field is inlined verbatim from `prompt.md` into the rendered workflow (per wake-provider/SKILL.md §2.3); for a Claude-based wake, that prompt prose IS runtime behavior — a model reading the prompt at execution time would see "MUST NOT be installed / not runnable" while the manifest's machine-readable `activation_state` says it IS runnable. **Lesson:** when flipping a manifest's `activation_state`, ALWAYS audit the prompt for matching state references — they're coupled. That clause now carries the lesson forward; future contributors flipping the state in either direction inherit the discipline. **Recommendation forwarded to γ-closeout (§7):** hoist "live-state-uniformly-applied" audit into the γ-scaffold template for any cycle that flips `activation_state`.

## §4 Live discovery (Stage 2) — the missing `status:review` label

### What happened

The wake fired on the responsive trigger ~3 minutes after cnos#491 was filed, claimed the cell per cnos#454 §2.2, invoked δ wake-invoked mode, drove γ/α/β to converge at R0 (zero findings), opened PR #492, then attempted the terminal `status:in-progress → status:review` lifecycle transition per cnos#454 §2.4 and the wake prompt's `output_contract`. **The cnos repo's actual label set did not contain `status:review`.** The available `status:*` labels were `status:todo`, `status:in-progress`, and `status:ready`. The wake applied `status:ready` as a fallback (semantically wrong — `status:ready` is pre-dispatch readiness per cnos.core/labels.json, NOT post-execution review) AND surfaced the discrepancy honestly in comment [`4784509561`](https://github.com/usurobor/cnos/issues/491#issuecomment-4784509561) ("Applied `status:ready` as the closest semantic equivalent. Operator may wish to: Create `status:review` label if the dispatch protocol's naming is canonical; OR Update the dispatch protocol and wake prompt to use `status:ready`. This is the first live cds-dispatch execution; the discrepancy is a live-discovery finding.").

The honest surfacing was the right behavior — the wake didn't pretend success. The silent label substitution was the wrong behavior — `status:ready` and `status:review` are doctrinally non-interchangeable.

### What was repaired

Operator-equivalent action (during bootstrap-δ session) — the canonical `status:review` definition was confirmed in `src/packages/cnos.core/labels.json` (color `5319e7`, description "Cell complete; awaiting external/operator review of the receipt/result.", owner `cnos.core`, group `lifecycle`). Applied `status:review` to cnos#491 via `mcp__github__issue_write`; GitHub auto-created the label on first apply (color defaulted to `ededed` rather than canonical `5319e7`; this is a separate small drift tracked in cnos#493's AC3). Flipped cnos#491 to remove `status:ready` + add `status:review`. Final cnos#491 state: `dispatch:cell + protocol:cds + status:review` (terminal; AC5 truly satisfied).

The smoke doc was updated honestly in R1 (`b7e5e27d` on cycle/491): added §7 "Live discovery: missing status:review label" recording what was discovered, why the labels are non-interchangeable, the repair performed, and the future-doctrine learning recommendation; updated §4 AC5 to PASS-with-live-discovery-and-repair; added two rows to the §5 transition table (wake fallback applying `status:ready`; operator-repair flipping to `status:review`).

### Doctrinal verdict

`status:ready` and `status:review` are NOT interchangeable. `status:ready` is **pre-dispatch readiness** (the cell is ready to be claimed); `status:review` is **post-execution review** (the cell is complete; awaiting external/operator review of the receipt/result). The lifecycle states are at opposite ends of the dispatch path; substituting one for the other inverts the cell's meaning to any future observer (admin wake reader, dashboard, downstream cycle). Operator-repair restored the correct lifecycle invariant.

### Future learning

The dispatch wake should NOT silently substitute missing terminal labels. The right doctrine (per cnos#493's out-of-scope follow-up enumeration): report `dispatch_label_missing` AND move the cell to `status:blocked` with a reason comment naming the missing canonical label. Do NOT pick a semantically-different existing label as a fallback. The cell then becomes a label-doctor input (or operator-repair input); the wake re-fires after the label is materialized.

Tracked as **cnos#493** (P1; live-discovery follow-up; design-and-build mode). The issue's in-scope work: audit cnos repo label set vs `labels.json` canonical set; materialize missing labels with canonical colors + descriptions; correct `status:review` color from `ededed` → `5319e7`; design + ship `cn install cnos.core` (or `cn label-doctor`) that reads `labels.json` + creates/updates labels idempotently; CI guard that fails on drift. The issue's out-of-scope-but-filed-as-follow-up enumeration: dispatch-protocol skill amendment to add `dispatch_label_missing` failure class with the correct behavior.
