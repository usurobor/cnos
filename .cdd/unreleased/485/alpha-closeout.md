## §3. What went well in R0

R0 converged first round with zero findings. The design choices that earned that:

- **Selector-from-manifest pattern.** The dispatch-aware `if:` constructor reads `selector.include[]` via `jq -r ".selector.include[$j]" "$manifest_path"` inside a loop, constructing the `contains(github.event.label.name, '<label>') || …` OR-chain from manifest values. AC7's grep oracle (`grep -ciE 'protocol:cds|cdr|cdw|dispatch:cell|status:todo' cn-install-wake` returns 0) and β's empirical line-by-line read of the diff (per OG-1) both passed without remediation. No obfuscation-via-concat trickery was needed; the manifest-read shape is the natural shape.

- **Flag-with-WARNING override (OG-3 Option A).** Choosing a single visibly-named flag — over Option B's dual-manifest drift risk and Option C's environment-variable invisibility — meant the override mechanism is self-documenting. The flag name itself names what is bypassed; `--help` carries the TEST-ONLY annotation; every use prints `WARNING — --activation-state-override=<state> bypasses the manifest's declared activation_state=<declared>` to stderr; non-`live` override values still refuse. β's five-property audit found no gaps.

- **OG-2 explicit `event_name == 'schedule'` short-circuit.** The dispatch-shape `if:` clause is structured `schedule || (issues && contains-or-chain)` rather than just `contains-or-chain` — so schedule-triggered firings (where `github.event.label` is undefined) fire the job unconditionally, and only issue events are gated on selector membership. The renderer source has an inline comment block citing γ's OG-2 directive; the CI step 9 has an explicit `grep -qE "github.event_name == 'schedule'"` regression test. This pattern would have caused a 1-3 round iteration in a less-scaffolded cycle (schedule-suppression is the obvious regression mode); γ pre-naming the correct shape in scaffold §6 averted that entirely.

- **Per-CI-step audit table populated alongside the renderer changes, not deferred.** As noted in §2, populating the bash-e audit at α implementation time rather than punting to β meant the cycle/476 class-trap surface (`grep -c` returning exit 1 on zero matches under `bash -e`) was reasoned about while the steps were being written, not after they were already wrong. The new step 10 (AC5 refusal smoke) is the one step where intended-success is a non-zero exit (3); writing the audit at the same time as the step made it natural to choose `set -u` (without `-e`) + `|| rc=$?` capture, rather than blindly inheriting the `set -eu` pattern from neighbouring steps. β's independent re-audit agreed cell-by-cell.

- **Idempotent-write logic from cycle/476 generalizes cleanly.** The renderer's `if [ "$existing_sha" = "$rendered_sha" ]; then print "(unchanged)" else write` pattern (cycle/476) worked verbatim for the dispatch shape. AC6 idempotence passed without any new code.

## §4. What was friction

- **FN-1 sha drift in cnos#485 AC8 cost negligible time.** cnos#485 AC8 cites `47824628…` as the agent-admin golden sha256; γ scaffold §11 FN-1 verified at branch base that the actual value is `fa6b8c0c…` (drift caused by post-cycle-476 cleanup commits `c353d432`, `4b633bb2`, `7ab62cb9`). Because γ caught + documented the drift inline before I started, I never relied on the issue's stale value: I used γ's verified `fa6b8c0c…` from the scaffold, verified post-render that the AC8 invariant still held, and called out the inheritance in self-coherence §FN-α-5. No time lost. Lesson: γ pre-verifying sha-pinned invariants against filesystem state (the discipline γ surfaced in scaffold §11 FN-1) is the right pattern; α inherited it for free.

- **FN-α-1 cross-package manifest discovery: necessary, but mildly surprising.** The pre-existing manifest lookup assumed `CN_PACKAGE_ROOT/orchestrators/<wake-name>/wake-provider.json`. CI re-render check and inner-loop developer invocation both default `CN_PACKAGE_ROOT` to the renderer's own package (cnos.core), which only contains agent-admin. Rendering cds-dispatch (owned by cnos.cds) required either explicit `--manifest` everywhere or a sibling-package fallback under `src/packages/*/orchestrators/<wake-name>/wake-provider.json`. I added the fallback; β reviewed and accepted. **Was it necessary?** Yes — the alternative would have been threading explicit `--manifest` through the CI workflow and any future developer command line, which would have been a worse footprint. **Was the surprise reasonable?** Reasonably — the renderer's owning-package assumption was implicit in cycle/476 because only one wake existed; the assumption only became visible when a second package's wake needed rendering. **Future risk:** if two packages declare the same wake-name, the fallback would find the wrong one. None do today (agent-admin in cnos.core; cds-dispatch in cnos.cds); a `--package <pkg>` pinning flag would address this if it becomes real. Out of scope this cycle; noted for future.

- **AC oracle ambiguity at AC5 exit code.** γ scaffold §5 AC5 pinned exit 3 explicitly (consistent with cnos#485 issue body suggestion). γ scaffold §11 FN-2 then flagged "β can flag this as a finding if the operator wants a different code" — which was a small ambiguity I resolved by adopting exit 3 (per γ's pin) and documenting the rationale in `cn-install-wake`'s header (0/1/2/3 exit code table extended for 3 = activation_state refusal). β verified exit 3 in AC5 oracle without surfacing it as a finding. The ambiguity didn't cost time, but it's worth noting that "the oracle pins X but β may flag X" is a small fork in the discipline — γ-scaffold could resolve it ahead of α next time by pinning hard and naming the rationale.

- **γ scaffold's α prompt size: appropriate for a fresh-Agent-session contract.** The α prompt + scaffold ran ~6.5K words (per γ FN-8). On the one hand this is heavy; on the other hand, the bootstrap-δ "no chat state" discipline means α sees the scaffold and nothing else, so length is the right tradeoff against under-specification. The scaffold §6 was not over-specific (it offered three options for OG-3 rather than pinning one); it was not under-specific (it pinned the `if:` shape γ wanted, the renderer-source authority audit, the activation_state refusal shape). The one place I'd say it was *slightly* over-specific is the per-CI-step table format itself — γ specified the columns but the precise format ended up identical between α's audit and β's audit, which suggests the format is already well-pinned and could be hoisted to a shared β-skill template (per γ FN-6 and β FN-β-4).

- **What surprised me:** β converging first round with zero findings. I'd planned for one R1 iteration on something (the OG-3 mechanism choice was the most likely friction point; the cross-package fallback was a second). Neither surfaced as a finding. The β review was thorough — every AC re-run independently, every operator-named guardrail re-walked, every CI step re-audited — and still came back clean. The cleanliness is downstream of γ's scaffold quality (explicit oracles; verified sha invariants; pre-named correct `if:` shape; three options for OG-3 with rationale for each) more than of α's implementation discipline.

## §5. Lessons for future α prompts

Things worth pinning forward to the γ-scaffold template (out of scope for this cycle; observations for the parent session to triage):

- **The per-CI-step bash-e audit table format is the right discipline.** Five-column table (`# / step name / line range / command substitutions or pipelines / guarded? / bash -e exit on intended-success input / notes`). β's independent re-population agreed with α's audit cell-for-cell (β FN-β-2). The table format catches the cycle/476 3-round class-trap (`grep -c` returning exit 1 on zero matches) reliably. Future scaffolds should embed the table format directly (or `@include` from a shared β-skill) so cycles don't restate the columns. β FN-β-4 also names this; γ FN-6 anticipated it.

- **Three-option presentation of OG-3 (override mechanism) earned its keep in α's reasoning.** γ scaffold §6 guardrail #3 listed Options A/B/C with concrete rationale for each. I weighed all three explicitly before picking A; the reasoning was preserved in self-coherence §Design. β verified Option A worked and observed (FN-β-3) that "future γ scaffolds in this template family might pin Option A as the preferred mechanism … the open choice consumed α's reasoning time, β's review time, and the AC has converged on Option A anyway." I disagree mildly with β here: the open choice cost α maybe 5 minutes of weighing and 50 lines of self-coherence documentation, and the resulting Option A is now empirically validated. If γ pins Option A in future scaffolds, the rationale ("flag is self-naming; WARNING is auditable; single source of truth; bypass-`live`-only by construction") should travel with the pin. Better: γ pins Option A *with the β five-property check inlined* as the audit oracle, so future α writes against the audit.

- **The right structure for α's self-coherence file.** §Design / §AC-by-AC table / §Per-CI-step bash-e audit / §R0 (review-ready signal) / §Friction notes. β read every section without remarking on the structure; γ-scaffold-template-amendment could pin these section names as the canonical α-output shape. The §Design section's "Approach" + "Key decision" + "α-side guardrail-2 AC (explicit)" sub-sections each earned their place: the Approach narrative is the entry point; the Key decision call-out is where the OG-3 reasoning lives (otherwise it would have been buried); the explicit α-side AC for OG-2 is exactly the kind of "γ told α to surface this explicitly and α did" check that prevents schedule-suppression regressions.

- **Operator-named guardrails (OG-N) as a γ-scaffold section.** γ FN-6 anticipates this. The three OG-N this cycle (renderer-source authority; schedule-unconditional `if:`; visibly-test-only override) were each caught by β empirically because γ named them. AC-by-AC oracles alone would have left OG-1 (obfuscation-via-concat) and OG-3 (override discoverability) under-checked. The OG-N section between §AC oracles and §α-prompt is the right structural slot — future template work should land it.

## §6. Empirical findings for cycle/485 specifically

- **Renderer source LOC: 501 → 682 lines, +181 net (admin-shape only → admin + dispatch shape).** The dispatch surface is ~36% bigger than the original admin-only renderer. The growth distributes roughly: ~40 lines for `--activation-state-override` flag parsing + help-doc; ~50 lines for role-conditional `output_contract` validation + dispatch-required field validation; ~30 lines for the `activation_state` §3.10 refusal gate; ~10 lines for the `issues_labeled_selector_match` trigger arm; ~40 lines for the dispatch-aware job-level `if:` constructor (the jq-based selector.include loop is the heaviest piece); ~10 lines for cross-package manifest discovery. Both are dispatch-specific and have no plausible admin reuse path. **No copy-paste was needed.**

- **CI extension introduced new failure-mode surface area, but the cycle/476 bash-e discipline absorbed it.** Five new `run:` steps and five modified steps; the dangerous shape (intended-success-is-non-zero-exit at step 10 — AC5 refusal smoke) was correctly identified during α implementation and guarded with `set -u` + `|| rc=$?` capture rather than the neighbouring steps' `set -eu`. β's independent audit (β §Per-CI-step audit table; FN-β-4) detected no bash-e regressions. ## §7. Recommendations

For δ / γ to triage in future cycles:

- **Lift `--activation-state-override` to a generalizable `cn install-wake --force-from-declaration-only` (or equivalent) global flag once a second declaration-only manifest emerges.** Today only cds-dispatch is declaration-only; if cdr-dispatch / cdw-dispatch / future observer-shape providers come in declaration-only too, the per-flag mechanism stays. The flag is already shape-agnostic (it operates on any `activation_state` value); only the help-doc wording is dispatch-specific. A clean refactor would be: keep the existing flag name (operators already know it); broaden the help-doc wording to name all shapes that may declare `activation_state: declaration-only`; preserve the WARNING-on-every-use + bypass-`live`-only semantics. This is a 10-line documentation change, not a code-shape change.

- **Test the dispatch-shape renderer against a synthetic cdr-dispatch fixture (forward-looking).** The `cnos.cdr` package exists in the repo tree; its `orchestrators/cdr-dispatch/wake-provider.json` does not yet. Once it lands (future wave), the install-wake-golden CI workflow's structural-shape check should be extended to cover it, exactly as cycle/485 extended cycle/476's coverage. The cross-package manifest discovery (FN-α-1) is the enabling primitive — it walks `src/packages/*/orchestrators/<wake-name>/`, so cdr-dispatch falls in for free. The actual test would catch any selector/protocol/runtime drift in the third wake provider before it leaks.

- **Future γ-scaffold template amendments to consider (per §5 lessons):** hoist the per-CI-step bash-e audit table format into a shared `cnos.cdd/skills/cdd/beta/SKILL.md` so cycle scaffolds `@include` rather than restate (γ FN-6 / FN-8; β FN-β-4 also names this); pin γ-OG sections as a dedicated scaffold-template section between §AC oracles and §α-prompt; pin Option A (CLI flag + WARNING + bypass-`live`-only) as the default declaration-only override mechanism with the β five-property check inlined as the audit oracle.

- **Verify-cited-sha-against-filesystem-state discipline (γ FN-1 / α FN-α-5 / β FN-β-1) deserves a formal contract in the δ wake-invoked mode skill (Sub 5B / cnos#486).** The cycle/485 chain was: γ verified `fa6b8c0c…` at branch base; α inherited the verified value and re-verified post-extension; β re-verified at PR HEAD. Three independent verifications, all agreeing. The discipline produced reliable handoffs across all three roles. Sub 5B's δ wake-invoked contract should formalize "verify sha-pinned invariants against filesystem state before dispatching the next role" as an explicit step in the dispatch protocol.
