## Engineering-level observations

**What made this audit tractable:** every REMOVE/FOLD classification had a clean, pre-existing replacement to cite. The single FOLD row's four behavioral cases each mapped one-to-one onto an existing Go test that already exercised the same invariant against the real `transitions.json` rule table, so the audit's evidentiary burden was "confirm this test exists and passes," not "write new coverage to justify removal." This made AC2 (no guard removed without a cited live replacement) a verification task rather than a construction task, and kept the diff's blast radius to two shell scripts and one doctrine doc with no `.go`, `transitions.json`, or CI-workflow-shape changes.

**What would have made it harder:** if the invariant the folded bash predicate proved had not already migrated to the Go test suite (i.e., if #591/#593's mechanization had left a gap rather than fully superseding the guard), the FOLD would have required either declining to fold or writing new coverage first — turning a one-commit removal into a design-and-build sub-task in its own right. Separately, had the `cell_kind` seam's prose been ambiguous rather than consistently self-qualifying (each passage naming observation-only + non-enforcement + a specific deferred-to issue/phase together), AC4 would have required either an edit or an escalation rather than a documented negative result — the fact that all cell_kind-adjacent prose independently satisfied the same three-part test made that AC closeable without touching code.

This is a cross-cycle mechanical interaction between the artifact-heading convention this cycle's own `self-coherence.md` used and the ledger's section-matching logic; it was not introduced by this cycle's diff and did not require an α-side fix, but is recorded here as an observed pattern in the artifact-classification mechanism itself.
