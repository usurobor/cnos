## What was built

Phase 2 of the CDS/issues FSM authority flip (issue #569): the FSM evaluator (`cn issues fsm evaluate`) gained a guard-gated `--apply` flag that mutates GitHub status labels only when a rule's guards pass, and three worker-facing surfaces (`cds-dispatch/SKILL.md`, `delta/SKILL.md` §9.6, `dispatch-protocol/SKILL.md` §2.9) were updated so that δ *requests* the `status:in-progress → status:review` transition through the FSM rather than writing the label directly. Concretely:

- A new label-write primitive (`ghAddLabel`/`ghRemoveLabel`/`applyStatusLabel` in `fetch.go`) — no reusable GitHub label-write helper existed anywhere in the repo before this cycle, so this is new code following the existing `fetch.go` dependency-free `net/http` idiom, not a refactor.
- Two new declarative rules in `transitions.json`'s `in-progress` state: a positive rule proposing `target_state: "review"` when `REVIEW-REQUEST.yml` and matter (PR/commits) both exist, and an explicit blocked rule when a review is requested but no matter backs it — making empty-review structurally unreachable for the new transition path, not just for the pre-existing `review`-state rule.
- Prose sync across the three worker-facing skill files so no surviving text describes δ/the wake as writing `status:review` as its own direct act.

## Design decisions and why

**Guard combination for the new `in-progress → review` rule.** γ's scaffold offered a non-binding suggestion (`any_true: [review_request_present, pr_has_commits]`, `all_false: [run_active]`). I deviated on both axes and documented the reasoning in `self-coherence.md` §ACs/AC1 rather than silently diverging:

1. Dropped the `all_false: [run_active]` guard. The rule's primary caller (δ's own wake-invoked cycle, requesting its own transition via cds-dispatch step 7) fires *while its own workflow run is still executing* — gating on `run_active == false` would make AC2's central mechanism ("a worker requests its own transition") inert in its main use case. That guard exists on the pre-existing dead-in-progress reconciliation rules for a different purpose (an external reconciler noticing a run has already died); reusing it here would have been a category error, not a conservative choice. I placed the new rule ahead of the existing `run_active`-gated rules in evaluation order so a review request with matter fires regardless of run state.
2. Tightened `any_true` to `all_true: [review_request_present]` (hard gate) plus `any_true` over the matter guards. AC2's own wording is a conjunction ("ensure REVIEW-REQUEST.yml **and** matter exist"), not a disjunction — encoding it as `any_true` across both would have let `pr_has_commits` alone satisfy the rule without `REVIEW-REQUEST.yml`, which is looser than what the issue actually asks for and looser than the destination state's own validity rule (which requires the same evidence bar). Matching the two rules' evidence bars means the transition and the state it lands in agree on what counts as sufficient.
3. Added a dedicated blocked rule for "review requested, no matter" rather than letting that case fall through to the pre-existing fallback rules (which would have silently proposed `todo` or `delta_recovery` — the wrong signal for a worker that explicitly asked for review, and a signal that would have masked AC3 for this specific path).

Worth revisiting later: this guard combination assumes δ's request always arrives synchronously mid-run. If a future cycle introduces an asynchronous or externally-triggered review request (not from the wake's own in-flight run), the `run_active`-ordering choice here should be re-examined — it was correct for the one caller this cycle names, not proven for all future callers.

**404-tolerance for idempotent label removal.** `ghRemoveLabel` treats HTTP 404 (label already absent) as success rather than an error, while `ghAddLabel` does not get equivalent treatment (GitHub's add-labels endpoint is already idempotent for "already present," so there is no analogous benign-404 case there). This was γ's scaffold's own flagged implementation-contract-adjacent decision, not left implicit: without it, a manually-recovered state (operator already removed the old label) or a partially-applied prior `--apply` (crashed mid-mutation) would turn a legitimate retry into a hard failure, defeating the idempotence AC1 requires. Locked by a dedicated unit test (`TestApplyStatusLabel_ToleratesNotFoundOnRemove`) rather than left as an unstated assumption.

## What would be worth revisiting later

- **No live end-to-end test of `assembleLive`'s apply path.** Only the fixture-facts + fake-server-apply combination is exercised; the real live-fetch + live-apply round trip against an actual GitHub API shape has never been tested (this mirrors Phase 1's own coverage gap on the live GET path — not a regression introduced this cycle, but also not newly closed by it). A future cycle wiring an end-to-end smoke against a disposable test repo would close this.
- **I5 (skill-frontmatter-check) could not be run locally** — the `cue` CLI was absent in the sandbox both at α's and β's session. Both roles relied on a structural argument (diff hunks land well past frontmatter blocks) rather than a direct run. This has now held across two independent role sessions without incident, but the sandbox's persistent lack of `cue` is itself worth flagging as environment debt if it recurs across future cycles — γ may want to track whether this is worth a process fix (e.g. vendoring a minimal `cue` binary into the sandbox image) if it keeps forcing structural-argument fallbacks instead of direct runs.
- **The pre-existing I6 small-change/triadic classification asymmetry** (β's Finding 1) is not an α-side implementation gap, but α's own AC5 evidence language ("byte-identical... before this cycle's changes") should have been phrased more precisely — β's independent re-run (with `--exceptions`, the actual CI invocation) surfaced a discrepancy α's own un-flagged evidence framing glossed over. Noted as a self-critique for future cycles: run AC5's gate checks with the exact CI invocation (flags included), not a close approximation, and account for classification-transition effects when comparing "before" (no `.cdd/unreleased/{N}/` directory exists on main) against "after" (the directory now exists) rather than asserting flat byte-identity.
