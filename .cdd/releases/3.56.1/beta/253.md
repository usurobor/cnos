### Finding B — α §2.5b check 6 does not cover sibling-enumeration patterns

α's §2.5b check 6 (schema/shape audit) is scoped to "when introducing or changing a canonical form, verify (a) new form present across all relevant files AND (b) any superseded form removed." F1 was neither: it was "new row added to a table whose enumeration is restated in a sibling section." The check as currently written would not catch this pattern at authoring time.

Two remediation paths considered:

1. **Extend α §2.5b check 6** to cover "enumeration consistency across sibling sections of the same doc." Problem: this is hard to mechanize without overfit — "sibling section restating a fact from the edited section" is a judgment pattern, not a grep pattern. Risk of the check becoming vague enough to be noise.

2. **Do nothing** — rely on review §2.2.8 (which did catch it). The review-side check is already reliable, specific, and already in the active skill set. Adding an authoring-side version would duplicate coverage at the cost of α friction.

Disposition: **drop / no patch.** The review-side catch is economic and repeatable; the authoring-side extension would overfit. Noted for cross-cycle pattern watching only.

Triage: **drop** — single-cycle friction below recurring-pattern threshold for spec change.

### Finding C — tag push blocked by env (process friction, not code)

After release commit `7cda5b2` landed cleanly on `origin/main`, `git push origin 3.56.1` returned `HTTP 403` from the env's git proxy. 4 retries with exponential backoff (2s/4s/8s/16s) all failed. `git push origin refs/tags/3.56.1:refs/tags/3.56.1` with explicit ref syntax also failed. The MCP GitHub tool surface has `list_tags` / `get_tag` but no `create_tag`.

Consequence: release workflow (`.github/workflows/release.yml`, triggered by tag push matching `[0-9]*.[0-9]*.[0-9]*`) did not fire. No binaries built. No GitHub release artifact created. The release commit is on main; an operator needs to push `3.56.1` for the release workflow to run.

Pattern: CDD release skill §2.6 (`Tag and push`) assumes `git push origin <tag>` works. When the env refuses, there is no documented fallback. The MCP surface does not expose a `create_tag` primitive.

Triage: **no MCA available** — both candidate remediations are outside CDD / cnos scope:
- (a) Add `mcp__github__create_tag` to the MCP GitHub surface (infra change, not cnos spec)
- (b) Add release-skill fallback "if `git push` returns 403, try MCP create_tag" (cannot be written against a primitive that doesn't exist)

γ should note this as an env constraint visible to future β sessions running in the same sandbox context — a knowable friction, not an actionable one. Per §9.1 MCA rule ("won't repeat without a mechanism is not an MCA"), this is not filed.
